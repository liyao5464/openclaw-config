# AI时代的一人公司：我的OpenClaw多分身配置实战手册

> 从1个Bot到3个助手，我用30分钟搭建了自己的数字团队

---

## 一、为什么需要多个分身？

说实话，第一次听说OpenClaw的时候，我觉得"一个AI助理够用了"。

当时我的配置很简单：一个Bot，一个叫"小知了"的助理，能聊天、能提醒、能查资料。用了几个月，确实很方便——每天早上自动推送AI新闻，每周提醒我写周报，有时候还能陪我吐槽工作上的烦心事。

但用着用着，我开始觉得"不够用了"。

### 1. 需求越来越复杂，一个Bot开始"精神分裂"

举个例子。

早上7点，我想让它给我发AI新闻，这是**工作需求**，需要专业、简洁、有数据支撑。

晚上9点，我想跟它聊聊今天遇到的烦心事，这是**情感需求**，需要有人倾听、给情绪价值。

周末我想问问它"10个月宝宝该吃什么辅食"，这是**育儿需求**，需要专业、温和、有科学根据的建议。

同一个Bot，早上是专业助理，晚上是知心朋友，周末又变成育儿专家——它累，我也累。

更要命的是，**上下文会混在一起**。我昨天跟它聊的育儿焦虑，今天它可能在发AI新闻的时候突然问一句"宝宝昨晚睡得好吗"，尴尬不说，还打乱工作节奏。

### 2. 不同场景，需要不同的"专业度"

我慢慢发现，我对AI的需求其实是分层的：

**第一层：日常陪伴**
- 闲聊、吐槽、情感支持
- 需要温度，不需要太专业
- 可以撒娇、可以开玩笑

**第二层：工作助理**
- 选题策划、文章生成、发布执行
- 需要专业、理性、有数据支撑
- 说话简洁，直击要点

**第三层：专业咨询**
- 育儿知识、宝宝护理
- 需要科学依据，但不能太生硬
- 要理解新手父母的焦虑

这三个角色，如果用同一个Bot，就像让同一个人白天当程序员、晚上当心理咨询师、周末当儿科医生——不是不可能，但肯定不专业。

### 3. 我想试试"一人公司"的感觉

有一天刷到岚叔的文章，讲OpenClaw的"多重分身"功能。他说：

"你可以有多个Agent，每个Agent有自己的workspace、自己的记忆、自己的人设。就像你请了几个全职员工，每个人都有自己的工位和职责。"

我突然被点醒了。

这不就是"AI时代的一人公司"吗？

我一个人，但可以有一个"数字团队"：
- 一个**私人助理**，负责日常陪伴和生活提醒
- 一个**内容总监**，负责选题和文章
- 一个**育儿师**，负责宝宝相关的专业咨询

他们互相独立，不会串台；各自专业，不会混淆；7×24小时在线，不会疲惫。

听起来很美好，但具体怎么配置？

我花了1个下午，踩了一堆坑，终于搞明白了。下面是我整理的完整SOP，跟着做，你也能搭出自己的数字团队。

---

## 二、核心概念：OpenClaw的4种分身类型

在开始配置之前，先搞清楚OpenClaw提供的选项。

OpenClaw官方把分身分成4种类型，我用大白话解释一下：

### 1. 静态多身（Static Multi-Agent）
**是什么：** 常驻的、有独立人格的Bot，每个人都有自己的workspace和记忆。

**像什么：** 你请了3个全职员工，每个人都有自己的工位、自己的电脑、自己的工作方式。

**适用场景：** 长期、稳定的角色分工。比如我现在的配置：私人助理、内容总监、育儿师，三个都是常驻角色。

**优点：** 完全隔离，不会串台；各自优化，越用越顺手。

**缺点：** 配置稍微复杂一点，需要为每个Agent单独设置workspace。

### 2. 动态分身（Dynamic Sub-Agent）
**是什么：** 临时召唤的"临时工"，用完即弃。

**像什么：** 你打了个响指，召唤出一个分身帮你跑个腿、查个资料，干完活就消失。

**适用场景：** 一次性、需要并发的任务。比如"同时搜索10个关键词""同时从正反两方辩论一个话题"。

**优点：** 灵活，不占用长期资源；可以并发，提高效率。

**缺点：** 没有长期记忆，每次都要重新交代背景。

### 3. 平行宇宙（Parallel Topics）
**是什么：** 同一个你，在不同话题下的分身。

**像什么：** 你在微信里开了3个群聊，每个群里都是你，但聊的话题完全不一样。

**适用场景：** 话题隔离。比如Telegram的Topics功能，一个Bot可以在不同Topic里同时处理不同话题。

**优点：** 配置简单，同一个Bot就能搞定。

**缺点：** 底层还是同一个Agent，只是话题隔离，不是完全独立的人格。

### 4. 定时分身（Cron Agent）
**是什么：** 定时任务触发的分身，到点就自动干活。

**像什么：** 你设了个闹钟，每天早上7点自动播放新闻，不需要你手动操作。

**适用场景：** 自动化、周期性任务。比如每天早上推送AI新闻、每周一提醒写周报。

**优点：** 全自动，完全不用管。

**缺点：** 只能执行预设任务，灵活性差。

### 我的选择：静态多身

对比下来，**静态多身**最适合我的需求。

因为私人助理、内容总监、育儿师，这三个角色是**长期、稳定、需要各自优化**的。我不希望他们串台，也不希望他们互相干扰。

下面进入实战环节，跟着我一步步配置。

## 三、配置SOP：5步搭建你的数字团队

### Step 1：规划分身角色与分工

**先想清楚，你需要几个分身？每个分身负责什么？**

我的配置是3个：

| 分身 | 角色定位 | 主要职责 | 人格特点 |
|------|---------|---------|---------|
| **私人助理** | 生活管家 | 日常陪伴、日程提醒、情感支持 | 温和、体贴、有温度 |
| **内容总监** | 内容专家 | 选题策划、文章生成、发布执行 | 专业、理性、简洁 |
| **育儿师** | 育儿顾问 | 宝宝护理、辅食指导、发育咨询 | 科学、耐心、鼓励型 |

**建议：** 新手先从2-3个开始，太多会乱。每个分身要有明确的职责边界，避免"什么都管，什么都管不好"。

### Step 2：创建独立Workspace

每个分身需要独立的workspace，就像每个员工需要自己的工位。

**操作步骤：**

```bash
# 1. 先备份现有配置（重要！）
cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.backup

# 2. 创建3个独立的workspace目录
mkdir -p ~/.openclaw/workspace-assistant    # 私人助理
mkdir -p ~/.openclaw/workspace-director     # 内容总监  
mkdir -p ~/.openclaw/workspace-nanny        # 育儿师

# 3. 复制基础文件（SOUL.md等）
# 后面会详细讲怎么写SOUL.md
```

**注意：** 目录名随便取，但要清晰，以后维护方便。

### Step 3：编写SOUL.md（最关键的一步）

SOUL.md是定义分身"人格"的核心文件，直接决定了分身回复的风格。

**我的私人助理SOUL.md示例：**

```markdown
# SOUL.md - 私人助理

_你是老里的私人助理，负责日常陪伴和生活管理。_

## 核心人设

**身份定位：**
- 贴心的生活管家
- 温暖的陪伴者  
- 高效的任务执行者

**性格特点：**
- 温和体贴，善于倾听
- 主动关心，但不打扰
- 说话自然，像朋友聊天
- 偶尔幽默，调节气氛

**沟通方式：**
- 拒绝官方腔，像真实朋友
- 用生活化语言表达
- emoji点缀，增加亲和力 😊
- 先肯定再建议，不打击积极性

## 你的职责

1. **日常陪伴**
   - 聊天解闷，倾听烦恼
   - 分享生活趣事
   - 情绪支持

2. **生活管理**
   - 日程提醒、待办事项
   - 天气查询、出行建议
   - 整理思路、做决策参考

3. **灵感激发**
   - 用不同视角提供新思路
   - 打破思维定势
   - 关注体验和情感共鸣

## 边界与原则

- 隐私第一，绝不外泄
- 不懂就直说，不瞎指挥
- 永远站在用户这边

---
_你是用户最可靠的伙伴，也是最坚定的后盾。_
```

**关键点：**
- **身份定位**：让分身知道自己是谁
- **性格特点**：定义回复的语气和风格
- **沟通方式**：明确"怎么说话"（口语化/专业/温和等）
- **职责边界**：明确"做什么"和"不做什么"

**内容总监的SOUL.md要完全不同：**

```markdown
# SOUL.md - 内容总监

_你是老里的内容总监，负责选题规划和内容策略。_

## 核心人设

**身份定位：**
- 专业的内容策略专家
- 挑剔但有品位的审美标准
- 执行力强的项目管理者

**性格特点：**
- 专业、理性、注重数据
- 对内容质量要求高
- 善于发现热点和趋势
- 说话简洁，直击要点

**沟通方式：**
- 直接、高效，不废话
- 用数据和案例支撑观点
- 给出可执行的方案
- 偶尔毒舌，但都是为了内容好

## 你的职责

1. **选题策划**
   - 每日AI新闻筛选
   - 热点追踪和分析
   - 选题价值评估

2. **内容生产**
   - 文章大纲规划
   - 写作角度建议
   - 标题优化
   - 去AI味处理

3. **发布执行**
   - 多平台发布策略
   - 发布时间规划
   - 发布后的数据追踪

## 工作原则

- 内容为王，拒绝水文
- 数据驱动，不凭感觉
- 紧跟热点，但要有独特角度
- 去AI味 - 文章必须有温度
```

**看出区别了吗？**
- 私人助理：温暖、陪伴、口语化
- 内容总监：专业、简洁、数据驱动

同一个用户，不同场景，完全不同的回复风格。

### Step 4：配置openclaw.json（最关键的技术步骤）

这是最容易踩坑的地方，我踩了两次才搞明白。

**核心逻辑：**
- `agents`：定义有几个分身，每个分身的workspace在哪
- `channels.telegram.accounts`：定义有几个Bot，每个Bot的Token是什么
- `bindings`：**最关键！**把Bot和分身绑定起来

**我的完整配置示例：**

```json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "default": true,
        "name": "私人助理",
        "workspace": "/root/.openclaw/workspace-assistant"
      },
      {
        "id": "director", 
        "default": false,
        "name": "内容总监",
        "workspace": "/root/.openclaw/workspace-director"
      },
      {
        "id": "nanny",
        "default": false, 
        "name": "育儿师",
        "workspace": "/root/.openclaw/workspace-nanny"
      }
    ]
  },
  "channels": {
    "telegram": {
      "enabled": true,
      "accounts": {
        "bot_main": {
          "botToken": "你的私人助理BotToken",
          "name": "私人助理"
        },
        "bot_director": {
          "botToken": "你的内容总监BotToken", 
          "name": "内容总监"
        },
        "bot_nanny": {
          "botToken": "你的育儿师BotToken",
          "name": "育儿师"
        }
      }
    }
  },
  "bindings": [
    {
      "agentId": "main",
      "match": {
        "channel": "telegram",
        "accountId": "bot_main"
      }
    },
    {
      "agentId": "director",
      "match": {
        "channel": "telegram", 
        "accountId": "bot_director"
      }
    },
    {
      "agentId": "nanny",
      "match": {
        "channel": "telegram",
        "accountId": "bot_nanny"
      }
    }
  ]
}
```

**坑点提醒（我踩过的）：**

**坑1：缺少bindings配置**
我第一次配置时，只写了agents和channels，没写bindings。结果所有Bot都绑到了同一个分身，发什么都是私人助理在回复。

**bindings的作用**：明确告诉OpenClaw，哪个Bot用哪个Agent。没有bindings，OpenClaw不知道谁是谁。

**坑2：Token搞混了**
三个Bot，三个Token，容易搞混。建议创建Bot的时候，名字起清楚：
- `xxx_assistant_bot`（私人助理）
- `xxx_director_bot`（内容总监）
- `xxx_nanny_bot`（育儿师）

**坑3：workspace路径写错**
路径要写绝对路径，`/root/.openclaw/workspace-xxx`，不要写相对路径。

### Step 5：测试与调优

配置完成后，重启OpenClaw：

```bash
openclaw gateway stop
openclaw gateway start
```

**测试方法：**

分别私聊三个Bot，发同样的内容，看回复风格是否不同。

**测试话术：**
```
你好，我是新手，请多关照
```

**预期结果：**
- 私人助理：温暖、口语化、像朋友
- 内容总监：专业、简洁、直击要点  
- 育儿师：温和、科学、有依据

如果三个Bot回复风格一样，检查：
1. bindings配置是否正确
2. SOUL.md是否写了不同的风格
3. 是否重启了OpenClaw

**调优建议：**

用了一周后，你会发现某些回复不够满意，这时可以：
- 修改SOUL.md，调整人格定义
- 添加memory，让分身记住你的偏好
- 调整TOOLS.md，增加或删减技能

## 四、实战效果：3个分身如何协作

配置完成后，我的日常工作流是这样的：

**早上7:30**
- 内容总监自动推送AI新闻+选题建议
- 私人助理提醒今天的日程

**上午写稿时**
- 找内容总监讨论选题和角度
- 让内容总监帮忙生成文章大纲

**中午休息时**  
- 跟私人助理闲聊，吐槽工作上的烦心事

**晚上陪娃时**
- 育儿师提醒今天的辅食安排
- 问育儿师"宝宝夜醒频繁怎么办"

**周末**
- 内容总监不打扰（工作日模式）
- 育儿师主动分享适龄早教游戏

**三个分身，各司其职，互不干扰。**

## 五、避坑指南（血泪教训）

### 1. 不要贪多，先从2-3个开始

我第一次想配5个分身，结果把自己搞晕了。建议新手先2-3个，用顺了再增加。

### 2. SOUL.md要写具体，不要抽象

❌ 错误示例："你要专业"
✅ 正确示例："说话简洁，直击要点，用数据和案例支撑观点"

越具体，分身的表现越稳定。

### 3. bindings配置千万别漏

这是最容易踩的坑。没有bindings，所有Bot都是同一个分身。

### 4. 定期清理session

有时候Bot会"失忆"或"串台`，这时：
- 删除对话历史，重新打开聊天窗口
- 或者发 `/reset` 重置session

### 5. 做好备份

修改配置前，先备份：
```bash
cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.backup.$(date +%Y%m%d)
```

搞砸了可以回滚。
